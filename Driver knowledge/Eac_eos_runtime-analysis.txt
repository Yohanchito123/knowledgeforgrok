EasyAntiCheat EOS Driver - Runtime Analysis

Follow-up to my static analysis post. Dumped the driver from kernel memory and dug through what's actually happening at runtime. This covers some of the detection mechanisms I mentioned I'd look into.

I will be live debugging the driver in the next few days, but this is just a runtime dump to see what is different.

One thing worth noting upfront - the runtime dump is a different build than the static binary I analyzed before. Different hash, different size (38MB vs 23MB), and different crypto parameters. EAC ships multiple variants, so your mileage may vary.


Binary Differences

The runtime dump came out significantly larger:

Code:
Static:  23.3 MB, SHA256: 7e7f9cfd...
Runtime: 38.4 MB, SHA256: 26da7d4b...
The extra ~15MB is mostly the virtualized section unpacking at runtime. Also found an extra segment (seg004, 24KB) that wasn't in the static binary. The virtualized section jumped from 21MB to 35MB once loaded.

The hardcoded modulus for API resolution is completely different:

Code:
Static:  0x199F07D3B49761CB
Runtime: 0x8123AD9AFB36519
So if you're trying to use specific values from someone else's analysis, make sure you're on the same build.


API Obfuscation (Confirmed)

The RSA-style resolver I described in the static analysis is confirmed. Exponent is still 17 for me, dual modulus system still applies:

Code:
// Primary resolver at sub_FFFFF80251340000
uint64_t resolve_api(uint64_t* context) {
    // Low 32 bits: base^17 mod runtime_modulus
    uint32_t low = modexp(*context, 17, qword_FFFFF8025150D3B8);
 
    // High 32 bits: base^17 mod 0x8123AD9AFB36519
    uint32_t high = modexp(value, 17, 0x8123AD9AFB36519);
 
    return ((uint64_t)high << 32) | low;
}

Detection Mechanisms

I found several detection systems working together.


Memory Scanning (sub_FFFFF802513A7174)

Page-by-page memory scanner with validation:

Code:
uint64_t scan_memory(uint64_t addr, size_t size, void* output) {
    uint64_t page_base = addr & ~0xFFF;
    uint32_t pages = (size + 0xFFF) >> 12;
 
    for (uint32_t i = 0; i < pages; i++) {
        if (MmIsAddressValid(page_base + (i << 12))) {
            void* mapped = map_physical_page(...);
            memcpy(output, mapped + offset, copy_size);
            unmap_page(mapped);
        }
    }
    return bytes_copied;
}

Page Table Walking (sub_FFFFF802513E6768)

This one's interesting. They bypass standard APIs entirely and walk page tables directly:

Code:
bool walk_page_tables(uint64_t virt_addr, uint64_t* phys_out, uint64_t* flags_out) {
    uint64_t cr3 = __readcr3();
 
    // PML4 -> PDPT -> PD -> PT
    if (read_physical((cr3 & 0xFFFFFFFFF000) + ((virt_addr >> 36) & 0xFF8), 8, &pml4e) == 8
        && (pml4e & 1)) {  // Present bit
 
        if (read_physical((pml4e & 0xFFFFFFFFF000) + ((virt_addr >> 27) & 0xFF8), 8, &pdpte) == 8
            && (pdpte & 1)) {
 
            // Handle 1GB/2MB large pages via PS bit
            // Continue to PT level for 4KB pages
            // ...
        }
    }
 
    *phys_out = physical_address;
    *flags_out = combined_flags;  // NX, Write, User
    return success;
}
This catches hidden memory that MmIsAddressValid wouldn't see. If your page tables show something that the API says doesn't exist, you're flagged.

Decrypted all five patterns. Here's what came out:

P1: 5da72a02d22a8963cd369666dfc65eaa7c5e28918e7cb7f17c1115a7c53b
P2: d7712c1f0fd5588ca70b66939ff791bbf7412ddb

The encrypted data in the binary is stored as hex-encoded bytes in UTF-16LE with null padding. When you decode the hex before XOR, you get weird pair-swapped sequences like `CBEDGFIHKJ` (P5) which unswaps to `BCDEFGHIJK` - literally sequential ASCII 0x42-0x4B.

The comparison function (sub_FFFFF80251391D38) does case-insensitive wide-string matching with `| 0x20`, but since the decrypted output is binary garbage (0xA75D, 0x022A, etc.), this is matching against binary data interpreted as UTF-16LE, not human-readable paths.

My best guess: these are PE header signatures, code fingerprints, or embedded watermarks from known cheat tools, not filename/path blacklists like I originally assumed (correct me if I'm wrong). The actual path blacklist might be elsewhere, or this version uses hash-based detection instead. Most likely the former.

What Gets You Flagged

Based on the detection functions I saw:

Immediate Detection:
Path/name matches blacklist substring
Driver code hash mismatch (patches, hooks)
Embedded PNG/files in process memory
Hidden memory (page table vs API mismatch)
Writable code sections or executable data sections

Suspicious/Logged:
Non-standard PE section names
Unusual process context patterns
Cross-process memory access patterns
Registry access to sensitive keys

This is just SOME of the things that get you flagged, there is more I just need to keep reversing.

SHA-1 integrity checking on module code sections
